## 開場：開發者如何在低程式碼與無程式碼中保持掌控？

多年來，低程式碼與無程式碼工具一直被視為「專為非開發者設計」的工具而遭到忽視。

如今，隨著這些平台的能力日益強大——具備資料建模、權限系統與基於插件的擴展——加上 AI 以爆炸性的速度進展，我們正進入一個全新的技術時刻。

AI 比以往任何時候都更快地接管重複性的程式碼撰寫工作。

💡 延伸閱讀：[GitHub 星數最高的 20 個開源 AI 專案](https://www.nocobase.com/en/blog/github-open-source-ai-projects)

大型語言模型正成為初階的程式碼生成器，能夠直接產出元件與基本邏輯。在此背景下，低程式碼與無程式碼平台已不再是單純的拖放式建構工具。它們已成為結構化、可治理的介面，用以與 AI 協作。

這些平台提供了清晰的架構邊界、預先定義的配置模型，以及可控的執行環境，讓 AI 能夠高效且安全地整合：

*   AI 驅動的商業邏輯：AI 可以直接在這些平台內建構複雜的工作流程或生成資料模型。
*   開發者的新角色：開發者不再需要花時間在 CRUD 工作上，而是可以專注於平台架構、擴展設計，以及 AI 無法處理的複雜整合與底層調校。

這為開發者帶來了新的問題：

隨著 AI 與低程式碼/無程式碼加速融合，我們應該在哪裡劃下程式碼的界線？我們該如何平衡速度與掌控，並確保長期的系統治理？

本指南將協助技術領導者與開發者清楚重新定義，何時採用低程式碼與無程式碼才是正確的選擇。

## 何時該使用？

以工程紀律評估適用性。如果一個核心系統符合任何「不適用」的條件，你應立即轉向傳統開發。


| 步驟                           | 檢查項目                                                                                                                       | 結果                |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| 步驟 1：業務結構     | 業務規則能否透過表格與工作流程圖清晰地建模？                                                        | 否 → 不適用     |
| 步驟 2：互動複雜度 | 介面是否需要超過中等複雜度（表單、表格與標準視圖）？                                      | 是 → 不適用    |
| 步驟 3：效能需求      | 系統是否需要即時回應（延遲 < 100 毫秒）、高並發、高吞吐量或底層效能調校？ | 是 → 不適用    |
| 步驟 4：擴展邊界   | 未來六個月的需求與擴展點是否可預測且模組化？                                               | 否 → 謹慎使用 |
| 步驟 5：團隊治理        | 團隊是否願意採用基於平台的工作流程並執行配置治理？                                        | 否 → 不適用     |

## 何時效果最佳

無程式碼與低程式碼的優勢在於，它們能將快速變動的業務元素（資料、工作流程、權限）與系統的穩定基礎（執行時期與渲染引擎）分離開來。

*   具有明確規則的清晰商業邏輯：圍繞資料模型、表單、工作流程與權限建構的系統。典型範例包括管理後台、內部審批流程、資料儀表板、工單工具與輕量級 CRM。
*   時間緊迫的小型團隊：非常適合內部工具，這類工具更重視可用性與可維護性，而非像素完美的設計。
*   跨職能協作：開發者負責底層架構與擴展（自訂 API、複雜邏輯），而業務與營運團隊則管理介面與工作流程調整。

## 何時不適用

在以下情境中使用無程式碼/低程式碼，往往會使抽象層變成效能瓶頸與架構黑盒子。

1.  **核心引擎與高需求工作負載**

*   高並發或即時任務，如交易引擎或串流處理，需要對 I/O、記憶體與演算法進行毫秒級的調校，這是平台開銷無法支援的。
*   大量運算，如 AI 推論或媒體處理，需要底層工程存取權限與不受限制的執行環境。

2.  **進階的前端互動性與使用者體驗需求**

大規模的消費者應用程式、複雜的自訂動畫或多裝置統一體驗，需要完整前端框架的靈活性。

3.  **反覆觸及框架限制的專案**

典型的陷阱是「你輕鬆完成了 80%，但缺失的 20% 正是產品的核心。」

這會導致無止盡的變通方案、二次開發，最終累積成巨大的技術債。

## 開發者保持掌控的五項原則

在使用無程式碼或低程式碼平台時，開發者應謹記：你的角色不是單純配置工具，而是設計它、治理它、擴展它。

1.  資料模型優先，而非介面

開發者必須掌握資料建模、關係設計與權限邊界。業務團隊可以建構 UI，但系統的基礎在於其資料結構與服務邊界。

2.  用平台處理結構化部分，在關鍵處撰寫程式碼

讓低程式碼/無程式碼處理可重複、可配置的工作。對於任何複雜、不可配置或涉及整合的部分，則使用自訂程式碼。

3.  在支援的邊界內擴展，避免使用駭客手法

遵循平台的擴展模型，並將自訂邏輯保留在可維護的位置。避免直接修改資料庫或繞過渲染層，這會讓未來的更新與維護變得痛苦。

4.  將配置視為工程資產

低程式碼/無程式碼仍然需要 DevOps。使用版本控制、環境提升（開發/測試/正式環境）、審批與回滾機制，以確保配置可追蹤且可控。

5.  建立共享能力，避免單點依賴

確保整個工程團隊都了解平台的架構、擴展點與治理規則。不要讓系統知識集中在少數人身上。

## 結論

無程式碼、低程式碼與 AI 不會取代開發者——它們只是改變了工程時間的運用方式。

讓平台接管重複性、結構化的工作，並將複雜且關鍵任務的部分保留在程式碼中。

核心原則保持不變：建構一個穩定的架構，以實現持續的業務敏捷性。
