これは **V2 体験を中心とした** 重要なアップグレードです。このバージョンでは **マルチポータル**、**新モバイル版**、**新コメントブロック** が追加され、同時に AI ナレッジベースとワークフローの機能も引き続き強化されています。すでに V2 を利用しているチームにとって、より注目すべきなのは全く新しい `/v/` ブランチです。これは V2 向けの独立したフロントエンドの入口として、今後の機能拡張を担っていきます。

## 新機能

### `/v/` ブランチの追加

`/v/` は全く新しい独立したフロントエンドの入口で、独立してビルドされます。2.0 バージョンではフロントエンドに対して非常に大きなリファクタリングを行いましたが、当時は互換性のために V1 のモジュールやページなどへの互換性を残していました。一方 `/v/` ブランチでは V1 の歴史的な遺産を完全に取り除き、V1 のページやブロックの作成・レンダリングはできなくなり、V2 向けの機能のみを残しています。同時に `/v/` ブランチはパフォーマンス面でも大きく最適化されており、このブランチは NocoBase チームが今後主に反復開発していくブランチになります。

> 本番環境がすべて v2 Page を中心としたユーザーであれば、`/v/` ブランチの利用を検討することをおすすめします。今後の新機能の反復開発をそのまま利用できるようになります。
>
> v2 Page を中心としつつも、少数の v1 Page があるユーザーは、まず v1 Page を v2 Page に移行してから、`/v/` ブランチの利用を検討するとよいでしょう。
>
> v1 Page を中心とするユーザーには、`/v/` ブランチへ直接切り替えることはおすすめしません。まず現在の v1 Page を v2 Page に移行してから、`/v/` ブランチへ切り替えることをおすすめします。

通常は既存のパスの前に `/v/` プレフィックスを付けるだけで、`/v/` ブランチに入ることができます。

例えば：

```
# 元のパス
https://nocobase.example.com/admin/1xl0epqn6q3

# v ブランチのパス
https://nocobase.example.com/v/admin/1xl0epqn6q3
```

#### アプリケーションのデフォルト入口の設定

環境変数 `APP_CLIENT_ENTRY_MODE` を使って、アプリケーションの入口モードを制御できます：

* `legacy-default`：`/` と `/v/` の両方にアクセスでき、デフォルトの入口は `/`
* `modern-default`：`/` と `/v/` の両方にアクセスでき、デフォルトの入口は `/v/`
* `modern-only`：`/v/` のみアクセス可能

#### `/v/` ブランチの現状

`/v/` ブランチは段階的な移行のためのブランチです。`/v/` ブランチに入ると、いくつかの明確な変化があります：

1. Page V1 で作成したページは、`/v/` ブランチでは表示されなくなります。一方、Page V2 で作成したページは、`/v/` ブランチでも正常に表示されます。
2. `client-v1` のみに対応したプラグインは、`/v/` ブランチでは読み込まれなくなります。`/v/` ブランチは `client-v2` に対応したプラグインのみを読み込みます。詳しくは[クライアントプラグイン開発ドキュメント](https://docs.nocobase.com/ja/plugin-development/client/)を参照してください。

現在、NocoBase に組み込まれている大部分のプラグインは、このバージョンで client-v2 への対応と移行を済ませているため、大部分の機能はそのまま利用できます（まだ移行していないプラグインが少数残っていますが、継続的に移行していきます）。ただし、一部のプラグインが提供する機能は、今後 `/v/` ブランチでのみ更新・反復されます。具体的には以下の通りです：

* プラグイン一覧に、廃止されたプラグインおよび 1.x のみ対応のプラグインは表示されなくなります
* Page V1 の新規追加はサポートされなくなります
* 公開フォームは v1 と v2 で区別され、`/v/` ブランチでは v1 バージョンの公開フォームは表示されません
* 承認ワークフローを `/v/` ブランチで使用する場合は、関連する UI を再設定する必要があります
* 新モバイル版は `/v/` ブランチのみ対応します
* マルチポータルは `/v/` ブランチのみ対応します
* App supervisor におけるサブアプリケーションの Display 設定は削除され、ポータルで代替できます
* 「AI 社員 / データソース」タブは削除されました
* コメントテーブル、カレンダーテーブル、式テーブル、外部データテーブルを削除しました
* ワークフローの手動ノードは、今のところ `/v/` ブランチには未対応です
* 今のところ、メニューへのバッジ数の追加には未対応です

本番環境の実際の状況に合わせて `/v/` ブランチへの移行を進めることができます。

### マルチポータル（`/v/` ブランチのみ対応）

ポータル（Portal）は、同一のアプリケーション内で複数のアクセス入口を提供するためのものです。各ポータルは、独立したページ、メニュー、ナビゲーション構造、レイアウト、権限設定を持つことができます。（Professional 版以上で対応）

実際の業務シーンでは、役割ごとに異なる操作画面を必要とすることがよくあります。例えば、ある小売管理システムでは：

```
小売管理システム

├─ 本部ポータル
├─ 店舗ポータル
├─ 販売代理店ポータル
└─ モバイルポータル
```

本部スタッフが注目するのは：

* 商品管理
* 在庫管理
* データ分析

店舗スタッフが注目するのは：

* レジ
* 棚卸し
* 注文処理

販売代理店が注目するのは：

* 仕入れ
* 照合
* 出荷状況

同じシステムを使っていても、役割ごとに同じメニューやページを見る必要はありません。マルチポータルはまさにこの問題を解決するためのものです。

![](https://static-docs.nocobase.com/2026-07-10-08-06-15.png)

参考ドキュメント：

* [マルチポータル](https://docs.nocobase.com/ja/multi-app/multi-portal/)

### 新モバイル版（`/v/` ブランチのみ対応）

新モバイル版は全く新しい `/v/mobile` 入口を提供します。それと同時に、ポータルでもモバイル版レイアウトのポータルを新規追加できるようになりました。

モバイル版レイアウトとデスクトップ版レイアウトは、同じデータソースと業務データを使用しますが、メニュー、ルーティング、ページの内容はそれぞれ個別に設定します。これにより、デスクトップ版のページ構造に縛られることなく、モバイルでの使い方に合わせてページを再構成できます。また、ボトムタブバーを第一階層のナビゲーションとして使用することで、スマートフォン上でのデータ入力、検索、承認、タスク処理により適した形になっています。

![](https://static-docs.nocobase.com/2026-07-10-08-04-59.png)

参考ドキュメント：

* [モバイル版レイアウト](https://docs.nocobase.com/ja/interface-builder/ui-layout/mobile)

### 新コメントブロック

新コメントブロックは、メインデータソースのコメントテーブルに限定されなくなり、関係ブロックでなくても使用できます。コメントブロックは、業務レコードにコメント機能を追加するためのものです。タスク、記事、チケット、顧客などのレコードの詳細ページやポップアップにコメントブロックを追加することで、ユーザーは現在のレコードを中心にコメントの閲覧、返信、新規追加ができます。

![](https://static-docs.nocobase.com/Comments-07-01-2026_12_02_PM.png)

参考ドキュメント：

* [コメントブロック](https://docs.nocobase.com/ja/interface-builder/blocks/data-blocks/comment)

### AI ナレッジベースの最適化

このバージョンでは、AI ナレッジベースも `client-v2` バージョンへの移行を完了しました。また、ワークフローでナレッジベースのドキュメントを作成する際に、ナレッジベースドキュメントの key による上書きに対応しました。

![](https://static-docs.nocobase.com/20260617003505.png)

参考ドキュメント：

* [AI ナレッジベース](https://docs.nocobase.com/ja/ai-employees/knowledge-base/knowledge-base/)

### ワークフロー：データベーストランザクションノード

ワークフローにデータベーストランザクションノードが追加されました。これは一連のデータベース操作を同一のトランザクション内で実行するためのものです。複数ステップのデータ処理で「すべて成功するか、すべてロールバックするか」を保証する必要があるシーン、例えば注文作成後の在庫の引き当て、明細の書き込み、ステータスの更新などに適しています。

![](https://static-docs.nocobase.com/20260610205505.png)
