## **引言**

2025 年初，我感覺每天在 [Reddit](https://www.reddit.com/r/nocode/comments/1khjivw/been_vibe_coding_for_5_months_but_went_back_to/) 上都會看到同一個問題：*「Vibe Coding 會取代無程式碼開發嗎？」*

這股熱潮無所不在——人們興奮地分享著使用 Vibe Coding 的實驗成果，從小型功能到完整應用程式都有。似乎只要與 AI 對話，幾乎任何東西都能被打造出來。

但幾個月後，氣氛轉變了。

一位使用者發文表示：*「Vibe coding 了 5 個月，但我還是回到了無程式碼開發。」*

![Vibe coding 了 5 個月，但我還是回到了無程式碼開發。](https://static-docs.nocobase.com/1-2y2mn5.png)

原因很直接：AI 生成結果的不穩定性（幻覺）。除錯和修正錯誤令人筋疲力竭，長期維護更是令人卻步。

對於許多在真實專案中測試過 Vibe Coding 的人來說，結論很明確：無程式碼開發仍然是更穩定、更可控的選項。

所以，真正的問題或許不是 *哪個比較好*，而是 *何時該使用哪一個*。

## **如何選擇？**

我越來越確信 Vibe Coding 不會取代無程式碼開發。相反地，它們更像是兩條平行的路徑——各自有其最佳的使用場景。

**Vibe Coding** 最適合小型、快速的專案。

想像一個個人首頁、一個簡單的公用程式，或是一個想快速實現的點子——那些只需要快速生成、沒有複雜邏輯的東西。

**無程式碼開發**（超越簡單的網站建置器）則更適合複雜、結構化的應用程式。

例如企業 CRM、審批系統和資料管理平台——這些需要協作、基於角色的權限、資料一致性以及長期維護的應用程式。在這些場景中，無程式碼平台提供的穩定性和控制力至關重要。

更有趣的是，兩者之間的界線正逐漸模糊。

許多無程式碼平台現在都整合了 AI 功能，允許使用者透過自然語言生成頁面、欄位或工作流程——同時仍保留無程式碼框架的一致性。

因此，真正的決定因素在於您的專案階段、需求的複雜度以及團隊的資源。這些因素將決定 Vibe Coding、無程式碼開發，或是兩者的組合，哪一個才是最合適的選擇。

## Vibe Coding

### **Cursor**

![Cursor](https://static-docs.nocobase.com/2-g9rzlq.PNG)

**網站：**[https://www.cursor.sh](https://www.cursor.sh)

**概述**

Cursor 是一個專為 AI 編碼體驗打造的 IDE。它不僅僅是 Copilot 的包裝，更是代表「Vibe Coding」方法的旗艦工具之一。

**主要特色**

* **程式碼即對話的工作流程**：Cursor 超越了簡單的自動補全。您可以反白程式碼區塊，並透過對話直接編輯它們、產生提交訊息，甚至重構架構邏輯。與 Copilot 相比，它感覺更主動且更可控。
* **更強的上下文感知能力**：與 VSCode 插件不同，Cursor 能更全面地捕捉專案上下文。這使得它在處理長檔案或多模組專案時特別有效，能更好地理解整體結構。
* **內建版本回退**：每次 AI 生成的變更都會自動產生 diff，讓您可以一鍵回退——降低了 AI「幻覺」破壞程式碼的風險。
* **可重複使用的提示詞市集**：Cursor 正在建立一個專用的 AI 提示詞市集，類似於 VSCode 程式碼片段。開發者可以重複使用他人精心設計的提示詞來改善 AI 輸出。

**最佳使用案例**

* 快速原型開發，以快速測試某個想法是否可行（經典的 Vibe Coding 使用案例）。
* 重構或改寫現有程式碼，例如更新樣式元件或 API。
* 一個用於探索 AI 在編碼任務中能協助到何種程度的「沙盒」。

**限制**

* **無法自託管**：不適合需要本地部署的企業或注重隱私的專案。
* **需要人為判斷**：雖然 Cursor 能產生邏輯正確的程式碼，但它不會為您做出架構決策或處理效能最佳化。
* **技能斷層的風險**：過度依賴 AI 生成可能使開發者變成「提示詞操作員」，而非真正的系統設計師。

### **v0.app**

![v0.app](https://static-docs.nocobase.com/3-ntbi6k.png)

**網站：**[https://v0.dev](https://v0.dev)

**概述**

由 Vercel 建構的 v0 是一個 AI 驅動的 UI 生成器，讓您可以透過自然語言建立前端頁面。它是 Vibe Coding 領域中最接近「設計轉程式碼」願景的工具之一。

**主要特色**

* **即時 UI 生成**：只需描述您的需求——例如 *「一個帶有搜尋欄和結果列表的部落格頁面」*——v0 就會立即產生響應式、語義化的程式碼（使用 **shadcn/ui** + **Tailwind CSS**）。
* **程式碼優先，無黑箱**：輸出是標準的 React + JSX，完全開放且可控——沒有供應商鎖定或隱藏的執行時期依賴。
* **對話式迭代**：您可以透過提示詞來優化設計，例如 *「將導航欄背景設為黑色」* 或 *「新增一個訂閱按鈕」*。v0 在套用變更的同時保持風格一致。
* **無縫 Vercel 整合**：生成的頁面可以一鍵部署、匯入專案或與團隊分享——使工作流程對開發者特別友善。

**最佳使用案例**

* 原型、活動網站或登陸頁面，具有吸引人的設計但邏輯簡單。
* 生成 UI 草稿，讓開發者能專注於業務邏輯。
* 在現有前端專案中生成元件以加速開發。

**限制**

* **僅限前端**：v0 不處理複雜的後端互動。業務邏輯和資料繫結需要手動實作。
* **風格單一**：目前的輸出高度依賴 **shadcn/ui** 函式庫。雖然看起來現代，但可能不適合有嚴格自訂設計系統的專案。
* **團隊協作有限**：v0 仍主要面向個人開發者；尚未支援多人編輯、權限或版本管理等功能。

### **Lovable**

![Lovable](https://static-docs.nocobase.com/4-0hlhc5.png)

**網站：**[https://lovable.so](https://lovable.so)

**概述** Lovable 採用更具對話性的方法：您用自然語言描述需求，它就會為您生成一個完整的網站。

**主要特色**

* **精簡的「生成 → 發布」工作流程**：只需描述您的需求——例如 *「我需要一個帶有輪播橫幅的家政服務網站」*——Lovable 就會自動建構頁面佈局、文字內容、圖片樣式，甚至配色方案。
* **可選的 AI 代理**：除了基本的網站生成，Lovable 還提供數十個預先建置的 AI 代理，例如 *SEO 顧問*、*設計助手* 或 *文案生成器*，以涵蓋純開發之外的工作。
* **交付成品**：目標不是給您一個開發環境，而是一個可直接使用的網站。您可以匯出 HTML/CSS/JS 或直接託管它。
* **零學習曲線**：專為非技術使用者設計，體驗更接近「用於網站的 ChatGPT」，而非傳統的建置器。

**最佳使用案例**

* 處於早期階段的初創公司，透過快速建立網站或登陸頁面來驗證想法（例如預約頁面、部落格、活動網站）。
* 內容驅動的專案，如作品集、行銷頁面或課程介紹。
* 想要比傳統網站建置器更智慧選項的自由工作者和非技術創作者。

**限制**

* **AI 風格化的輸出**：生成的網站在設計和文案上通常帶有「模板化」的 AI 外觀——非常適合 MVP，但不太適合精緻的品牌網站。
* **無複雜邏輯**：Lovable 不支援資料建模、權限控制或後端工作流程等結構化功能，使其不適合內部工具或系統級應用程式。
* **受限於對話範圍**：一旦需求超出簡單規則——例如會員系統或動態表單——Lovable 很快就會達到極限。

## 無程式碼開發

### **NocoBase**

![NocoBase](https://static-docs.nocobase.com/5-x1k9fg.png)

**網站：**[https://www.nocobase.com](https://www.nocobase.com)

**概述**

一個以資料模型驅動、開源、AI 驅動的無程式碼平台，專為建構中大型內部系統而設計。憑藉其高擴展性和複雜的權限控制，它扮演著 Vibe Coding 無法取代的「業務級場景專家」角色。

**主要特色**

* **以資料建模為核心**：與以表格為中心的無程式碼工具（例如 Airtable 風格的平台）不同，NocoBase 遵循類似框架的方法。它強調實體關係、欄位結構以及業務邏輯的清晰分離——非常適合建構複雜的業務應用程式。
* **靈活的插件生態系統**：所有功能都模組化為插件——從使用者認證、基於角色的權限和工作流程，到檔案儲存甚至 AI 助手。團隊可以根據特定需求自訂其技術棧。
* **前後端分離 + 區塊式 UI**：像樂高積木一樣建構 UI，自訂元件或整合 API。結果是一個既靈活又可控的系統。
* **細粒度的權限**：具備欄位級和行級存取控制，搭配前端邏輯，可以輕鬆配置特定於角色的檢視——這是 Vibe Coding 工具無法實際實現的。
* **系統中嵌入的 AI 員工**：將 AI 能力無縫整合到使用者介面、業務工作流程和資料上下文中，使 AI 能夠在真實的企業場景中實際應用。

**最佳使用案例**

* **企業級內部系統**：CRM、工單系統、資產管理、審批平台。
* **自託管的業務平台**：適用於需要安全性、控制力和自訂功能，同時又不依賴雲端供應商的團隊。
* **結合無程式碼 + 程式碼的技術團隊**：使用無程式碼快速建構核心系統，同時保留自訂開發的擴展性。

**限制**

* **學習曲線較陡**：與 v0.app 或 Lovable 等輕量級工具相比，NocoBase 更適合具有一定開發或產品設計經驗的使用者。
* **非網站建置器**：不適用於內容密集型網站、品牌網站或簡單的前端專案。
* **插件生態系統仍在發展中**：作為一個開源專案，插件和模板的市集正在成長，但尚未達到商業平台的水平。

### **Retool**

![Retool](https://static-docs.nocobase.com/6-938ipz.png)

**網站：**[https://retool.com](https://retool.com)

**概述**

Retool 是一個用於建構企業級內部工具的全方位平台。其優勢在於「連接一切 + 快速啟動」，具有強大的資料整合和業務邏輯能力。

**主要特色**

* **輕鬆的資料和 API 連接**：Retool 原生支援數十種資料庫（PostgreSQL、MongoDB、Snowflake 等）和第三方 API（Stripe、Slack、Salesforce 等）。配置一次，然後拖放元件即可建構功能介面。
* **豐富且高度靈活的 UI 元件**：配備廣泛的生產級元件庫（表格、圖表、表單、日期選擇器等），支援狀態管理和邏輯繫結——非常適合建構業務就緒的前端。
* **強大的腳本撰寫能力**：除了視覺化建置，您還可以使用 JavaScript 進行資料轉換、驗證或元件間邏輯——非常適合複雜的工作流程。
* **AI 驅動的開發輔助**：
  * **查詢與程式碼生成**：在 SQL 或 JS 編輯器中使用提示詞來生成查詢和腳本，減少重複性工作。
  * **AI 元件生成器**：用自然語言描述一個頁面，Retool 就會自動建立它。
  * **LLM 整合**：內建向量資料庫（Pinecone、Weaviate 等）的連接器，並支援 OpenAI、Cohere、Anthropic——讓添加 AI 搜尋、問答或文件智慧功能變得容易。
* **企業級安全與部署**：SSO、審計日誌和自託管部署選項使 Retool 非常適合具有嚴格合規性或安全性要求的組織。

**最佳使用案例**

* 快速建構結構化的內部系統，例如 CRM、審批工作流程、BI 儀表板和營運面板。
* 建立由 AI 增強的業務工具——例如客戶支援後端或 AI 驅動的分析系統。
* 中小型開發團隊使用 Retool 代替從頭建構內部工具。

**限制**

* **價格高昂**：商業版本的價格可能很高，尤其是按使用者計費，這可能對初創公司或預算有限的團隊構成障礙。
* **非完全無程式碼**：雖然 UI 建置器是視覺化的，但進階邏輯幾乎總是需要 JavaScript，使其不適合沒有程式設計背景的使用者。
* **功能性而非美觀性**：設計傾向於實用性和效率。雖然非常適合內部工具，但不太適合需要精美美學的客戶面向 UI。

**網站：**[https://www.tooljet.com](https://www.tooljet.com)

**概述**

ToolJet 是一個開源的低程式碼平台，圍繞著 **拖放式 UI + 多資料來源整合** 建構。近年來，它也引入了 AI 功能，使其成為混合平台如何演進的一個有力範例。

**主要特色**

* **視覺化建置 + 廣泛的資料來源支援**：連接 MongoDB、PostgreSQL、MySQL、REST API 等十多種來源。透過視覺化編輯器將資料繫結到元件，快速建構儀表板、管理面板或客戶管理系統。
* **直觀的拖放式 UI**：配備豐富的元件集（表格、按鈕、圖表等），並允許自訂 JavaScript 腳本進行互動邏輯——降低了前端密集型專案的門檻。
* **AI 增強的開發**：支援基於提示詞的頁面和邏輯生成，為非開發者提供了更簡單的入門途徑。
* **開源 + 自託管**：完全開源，支援 Docker 部署，適合有嚴格安全和合規需求的企業或團隊。
* **基於角色的存取控制**：為不同使用者和角色提供細粒度的權限，實現中型專案的有效協作。

**最佳使用案例**

* 快速建構結構化、資料驅動的內部系統（CRM、CMS、審批工作流程）。
* 需要自訂功能但無需編寫完整前後端程式碼的技術團隊。
* 希望將 AI 嵌入業務 UI 並驗證產品想法的初創公司或 AI 團隊。

**限制**

* **AI 仍處於輔助階段**：主要協助查詢生成和程式碼片段；尚未提供像真正 Vibe Coding 工具那樣的「從頭建構應用程式」體驗。
* **複雜互動有限**：雖然支援 JS 腳本，但建構高度自訂的 UI 或管理進階狀態邏輯，與完全編碼或高度可程式化的框架相比仍然更困難。

### **Bubble**

![Bubble](https://static-docs.nocobase.com/9-6jhy5s.png)

**網站：**[https://bubble.io](https://bubble.io)

**概述**

Bubble 是歷史最悠久、最成熟的全端無程式碼平台之一，涵蓋從 UI 設計到資料庫邏輯的所有內容，全部採用完全視覺化的工作流程。近年來，它透過插件和 API 整合擴展以擁抱 AI，更接近 Vibe Coding 的場景。

**主要特色**

* **無需程式碼的全端開發**：Bubble 支援頁面設計、資料庫建模、工作流程自動化、使用者認證和插件整合——使其成為少數能夠在無需編碼的情況下進行端到端應用程式建置的平台之一。
* **高度視覺化 + 響應式設計**：拖放元素來建構頁面，同時配置互動邏輯（條件、狀態、動態資料）。非常適合需要前端和後端邏輯的網頁應用程式。
* **AI 能力**：
  * **Bubble AI 助手**：透過自然語言添加工作流程或 UI 元件，使開發更具對話性。
  * **GPT 插件生態系統**：快速整合 OpenAI API，用於內容生成、問答、表單自動填寫等。
  * **LangChain 與 RAG 整合（實驗性）**：一些進階使用者正在將 Bubble 用作 RAG 驅動助手的前端層。
* **強大的社群 + 模板市集**：數百個模板和插件可供重複使用和自訂，加速上市時間。

**最佳使用案例**

* 建構完整的 SaaS 應用程式、使用者入口網站、預訂系統或教育平台——特別是對於從 0 到 1 的初創公司。
* 建立結合強大 UI/UX 與資料工作流程的 MVP。
* 開發 AI 驅動的應用程式，例如問答機器人、AI 寫作工具或對話式助手。

**限制**

* **學習曲線比預期陡峭**：儘管是無程式碼，但 Bubble 的工作流程和權限設定對初學者來說可能很複雜。
* **自託管支援薄弱**：Bubble 是閉源的，僅作為託管服務提供，限制了其在企業或資料敏感專案中的採用。

### **Flowise**

![Flowise](https://static-docs.nocobase.com/10-0acu8o.png)

**網站：**[https://flowiseai.com](https://flowiseai.com)

**概述**

Flowise 是一個用於 LLM 驅動應用程式的開源視覺化建置器。它圍繞著 **拖放式 AI 元件** 設計，既可以作為無程式碼平台的 AI 擴展，也可以作為 AI 驅動業務工具的獨立引擎。

**主要特色**

* **Node-RED 風格的工作流程建置器**：透過將 LLM（OpenAI、Claude 等）、向量資料庫（Pinecone、Chroma、Weaviate）、文字處理節點和 API 輸入/輸出等模組鏈接在一起，組裝成管道——就像用積木搭建完整的 AI 應用程式。
* **AI 的後端邏輯層**：Flowise 不處理 UI。相反地，它提供 AI 應用程式背後的邏輯和工作流程層，您可以將其作為 API 暴露給任何前端（例如 Bubble、ToolJet、Retool）。
* **輕量級 + 自託管友善**：只需幾個指令即可執行，無需註冊或雲端依賴——非常適合內部測試和整合。
* **涵蓋核心 LLM 使用案例**：支援聊天機器人、RAG 管道、文件問答、摘要、JSON 生成和知識庫查詢——非常適合垂直領域的 AI 業務工具。

**最佳使用案例**

* 為現有的無程式碼/低程式碼平台添加 AI「智慧層」。
* 建構可重複使用的 AI 服務（例如 FAQ API、文件摘要器、語義搜尋 API）。
* 技術團隊實驗不同的 LLM 工具鏈，作為 **AgentOps** 工作流程堆疊的一部分。

**限制**

* **無 UI 層**：Flowise 不是一個端到端的無程式碼工具；它需要與前端平台搭配使用。
* **大規模工作流程的複雜性**：雖然拖放降低了門檻，但除錯大型基於節點的工作流程仍然需要對輸入/輸出有技術理解。
* **不適合初學者**：與 Bubble 或 Lovable 等「即插即用」工具相比，Flowise 更像是一個 **開發者工具包**，而非消費者友善的建置器。
